# Superpowers Handoff

What Fable built in Phase 0, and how to turn it into the full
`trading-office` application.

## What exists

- **`packages/office-visual-kit`** — reusable floor renderer:
  Tiled loader/normalizer, scene schema + validation, asset registry,
  tile/agent/object/badge renderers, interaction manager, pixi-viewport
  camera, React wrapper (`OfficeSceneCanvas`, `useOfficeScene`). Core is
  plain PixiJS v8; React is wrapper-only (acceptance criterion).
- **`examples/trading-lab-research-floor`** — runnable preview
  (`npm install && npm run dev` from the repo root → http://localhost:5173):
  7 agents (Boss/Orchestrator, Strategy Analyst, Researcher, Critic, Builder,
  Evaluator, Performance Monitor), interactive objects (Boss Console, Wall
  Monitor, Hypothesis Board, Bot Status Monitor, Archive Shelf, Server Rack),
  hover/click debug card, preview-side status simulation (off by default),
  and a **Day Office / Night Control Room theme toggle** (Day is the
  default). Workstations face the viewer: 2×1 desk blocks with an aluminum
  monitor (screen toward the agent, glowing mark on its back) and a
  front-facing seated agent behind the desk; role labels are nameplate
  plaques on each desk's front edge (`agentLabelOffsetY` + label
  `borderColor`; per-agent `labelOffsetY` for the Boss's deeper desk);
  badges sit just above heads, on a 4-row workstation pitch. The Boss is
  screen-centered behind a 4×2 mahogany command console (a sprite-less
  hit-area object over console tiles); a glass-walled infra room below him
  (square corners, sliding glass door) groups Server Rack, Archive and Bot
  Status over a tech floor.
- **Theme model** — both themes share one geometry; each theme is a generated
  tileset + a `.tmj` variant + a `Partial<OfficeSceneTheme>` block. Switching
  themes is a scene-config swap in the preview — no kit involvement.
- **Asset pipeline** — environment art (tiles, props) is original
  CC0-equivalent pixel art generated by deterministic scripts (`tools/`),
  drawn on a logical 16-px grid and upscaled ×2 for a chunky classic look:
  32×32 tiles (day art + derived night variant via `nightify()` with
  emissive screens/LEDs/monitor marks). The **agents are composed from
  real Universal LPC Spritesheet Character Generator layers**
  (`tools/compose-lpc-agents.mjs`: chair-sit pose, the generator's palette
  ramps, cropped to 64×52 seated busts, identical 2-frame strips — no idle
  animation; the Boss gets an original winged executive chair drawn behind
  the LPC figure). LPC sprites + SOURCE/ATTRIBUTIONS/LICENSES live under
  `public/assets/third-party/lpc/` (CC-BY-SA 3.0); everything else stays
  under the MIT `LICENSE` at the repo root.
- **Map pipeline** — valid Tiled `.tmj` (generated by a reviewable script,
  editable in Tiled), conventions documented. The renderer is tile-size
  agnostic — 32×32 here is a floor choice, not a kit constant.

## How to run / verify

```bash
npm install
npm run dev        # preview at :5173 (Day Office default; toggle in the topbar)
npm run build      # typecheck + production build
npm run generate   # regenerate assets + maps (both themes, deterministic)
```

## Integration sketch for the production app

```text
apps/web (React 19)
  └─ <OfficeSceneCanvas config={floorConfig}
        onAgentClick={openAgentPanel}
        onObjectClick={(o) => openPanel(o.panelTarget)}
        onSceneReady={(scene) => sceneRef.current = scene} />

apps/server (office gateway)
  └─ pushes agent/workflow state → web → scene.setAgentStatus(id, status)
```

- `ObjectEntity.panelTarget` is the routing key for panels — the example
  already assigns `backtest-summary`, `hypothesis-pipeline`, `bot-health`,
  `knowledge-base`, `infra-status`, `boss-commands`, `shared-context`.
- Multiple floors = multiple `OfficeSceneConfig`s (one `.tmj` + config per
  agent system). The elevator/floor-switcher is app chrome, not kit code.
- The kit is consumed as TypeScript source via workspace exports
  (`.` and `./react`). If you need a publishable build, add a `tsc`/bundler
  emit — nothing in the source blocks it.

## Future real-time architecture (notes only — NOT implemented in Phase 0)

Design constraints to honor when live data arrives; none of this exists yet
and none of it should be built ahead of the gateway:

- **PixiJS is a renderer, nothing else.** It consumes imperative calls
  (`scene.setAgentStatus(...)`, future `scene.setObjectBadge(...)`); it never
  owns business state.
- **Live agent/workflow state lives outside React** — a future
  `OfficeRuntimeStore` (plain TS, subscribable) holds the canonical runtime
  state pushed by the gateway.
- **A future `RuntimeSceneBridge`** subscribes to the store and translates
  state diffs into the scene's imperative methods. One bridge per mounted
  scene; React renders neither agents nor badges.
- **React panels must not update Pixi objects every tick.** Panels read the
  store (e.g. `useSyncExternalStore`); the bridge — not React render cycles —
  touches the scene. Re-rendering a panel must never re-render the floor.
- **WS/SSE updates call imperative scene methods** via the bridge; no prop
  drilling of live state into `OfficeSceneCanvas`.
- **ParticleContainer effects, agent-to-agent link lines, walking animations
  are future work**, not Phase 0 — the depth-sorted container model supports
  adding them without restructuring.

## Intentionally missing (do not look for it)

- No backend, no Hono, no Postgres, no auth, no Docker Compose.
- No real `trading-lab` API client, no SSE/WebSocket — the status simulation
  in the preview is a `setInterval` stand-in for the future gateway stream.
- No production dashboard panels — debug card only.
- No execution authority anywhere in this repo (architecture rule:
  `trading-office` visualizes and routes commands; `trading-platform`
  executes).
- No drag-and-drop floor editor — Tiled is the editor.

## Recommended next steps

1. **apps/web shell**: floor switcher + panel router around
   `OfficeSceneCanvas`; replace the debug card with real panels keyed by
   `panelTarget`.
2. **Office gateway (apps/server)**: read-only adapters to `trading-lab`
   (mock mode first), stream agent statuses; wire to `setAgentStatus` and a
   future `scene.setObjectBadge`-style API if objects need live markers.
3. **Status vocabulary**: confirm the `AgentStatus` union against real
   trading-lab workflow states; extend `DEFAULT_STATUS_COLORS` accordingly.
4. **More floors**: author the next agent system as `.tmj` + scene config —
   that path is the kit's whole point and needs no renderer changes.
5. **Art pass**: replace generated placeholders role by role (see
   asset-guidelines.md); the registry isolates art from code.
6. **Perf, if floors grow**: tile layers are plain sprite containers; if a
   floor exceeds a few thousand tiles, add `cacheAsTexture` on static layers
   inside `renderTileLayers` — measure first.

## Known sharp edges

- `OfficeSceneCanvas` recreates the scene when `config` identity changes —
  memoize configs.
- The example's `main.tsx` skips `<StrictMode>` (double-mounted effects vs
  async Pixi init in dev); the wrapper cleans up correctly either way, but if
  you enable it, verify there's no canvas flicker in dev.
- Tile data must be CSV or uncompressed base64; embedded tilesets only —
  loader errors are explicit about both.
- `pixi.js`, `react` and the kit must be deduped to single versions in the
  final app bundle (the example's vite config shows `resolve.dedupe`).
- Agents are drawn **seated, facing the viewer** (chair-sit bust, the
  chair is baked into the sprite) — perfect for the workstation layout,
  but a future walking/animation system needs walk-cycle frames. Since the
  characters already come from the real LPC generator, the walk/idle/run
  sheets of the SAME layer combinations are one composer extension away
  (`tools/compose-lpc-agents.mjs` LOOKS table + the generator's `walk.png`
  sheets; attribution regenerates automatically).
- Theme switching remounts the canvas (`key={themeName}`) because the
  `<Application>` background color is an init option; runtime status state is
  reset on switch (statuses come from config / the future gateway anyway).
